## Task 1 詳細: 画像ソースパターン判定

### コード経路

```
reviseFixedPriceItem()
  └─ readListingDataFromSheet(spreadsheetId, rowNumber)  [line 1105]
       └─ images: extractImageUrls(rowData, headerMapping)  [line 443]
            └─ 画像1〜23列 + ストア画像列 を1つの配列にマージして返す
```

### `extractImageUrls()` の実装 (line 331-361)

```javascript
function extractImageUrls(rowData, headerMapping) {
  const urls = [];

  // 1. 画像1〜23 を全列スキャン
  for (let i = 1; i <= 23; i++) {
    const url = getValueByHeader(rowData, headerMapping, '画像' + i);
    if (url && String(url).trim() !== '') {
      const convertedUrl = convertDriveUrlForEbay(url.trim()) || url.trim();
      urls.push(convertedUrl);     // ← 画像列の値が全て入る
    }
  }

  // 2. ストア画像 を末尾に追加
  const storeImageUrl = getValueByHeader(rowData, headerMapping, 'ストア画像');
  if (storeImageUrl && String(storeImageUrl).trim() !== '') {
    const convertedUrl = convertDriveUrlForEbay(storeImageUrl.trim()) || storeImageUrl.trim();
    urls.push(convertedUrl);       // ← ストア画像も同一配列の末尾に追加
  }

  return urls;
}
```

### 判定: **パターンC（両列をマージ）**

- `readListingDataFromSheet()` の `listingData.images` には「画像1〜23」と「ストア画像」が無条件にマージされた配列が格納される
- パターンA「image列のみ」でも、パターンB「store image優先フォールバック」でもない
- `normalizeImagesForRevise()` には、このマージ済み配列がそのまま渡される

### Pattern C の Layer 6 との組み合わせ問題

`_detectImageDivergence()` は文字列の厳密一致で比較する（`.indexOf(u)`）。

**新規出品後にシートに書き戻されたEPS URLが維持されている場合**（正常ケース）:
- `sheetUrls` = `["i.ebayimg.com/PROD", "i.ebayimg.com/STORE"]`
- `ebayUrls`  = `["i.ebayimg.com/PROD", "i.ebayimg.com/STORE"]`
- → 差分なし、popup なし ✅

**画像列に Google Drive / lh3 URL が残っている場合**（過去の出品や書き戻し失敗）:
- `sheetUrls` = `["lh3.googleusercontent.com/d/XXX", "i.ebayimg.com/STORE"]`
- `ebayUrls`  = `["i.ebayimg.com/EPS_PROD", "i.ebayimg.com/STORE"]`
- → `lh3.googleusercontent.com/d/XXX` ≠ `i.ebayimg.com/EPS_PROD` → 差分検出 → popup
- この差分は「シートを更新しない限り永続」するため、毎回 popup が発生する

## 修正提案

### 問題1（重大）: `use_ebay` 選択時にシートが更新されない

**問題**: ダイアログ UI は「eBay側を維持して**シートに取り込む**」と表示するが、`resumeReviseAfterDivergence()` にシートへの書き戻し処理がない。そのため次回更新でも同じ差分が検出され、popup が再発し続ける。

**修正案**: `resumeReviseAfterDivergence()` の `use_ebay` 分岐に、eBay URLをシートの画像列に書き戻す処理を追加する。

```javascript
// resumeReviseAfterDivergence() 内の use_ebay 分岐に追記
if (choice === 'use_ebay') {
  try {
    // eBay側のEPS URLをシート列に書き戻す
    // extractImageUrls() と同じ列順 (画像1〜23, ストア画像) でセルを更新
    const ss = getTargetSpreadsheet(spreadsheetId);
    const sheet = ss.getSheetByName(SHEET_NAMES.LISTING);
    if (sheet) {
      const hMap = buildHeaderMapping();
      const colOrder = [];
      for (var i = 1; i <= 23; i++) colOrder.push('画像' + i);
      colOrder.push('ストア画像');

      var epsIdx = 0;
      for (var ci = 0; ci < colOrder.length && epsIdx < info.ebayUrls.length; ci++) {
        var col = hMap[colOrder[ci]];
        if (!col) continue;
        var existing = String(sheet.getRange(info.rowNumber, col).getDisplayValue()).trim();
        if (!existing) continue;
        sheet.getRange(info.rowNumber, col).setValue(info.ebayUrls[epsIdx]);
        epsIdx++;
      }
      Logger.log('層6: eBay URL をシートに書き戻し: ' + epsIdx + '列');
    }
  } catch (writeErr) {
    Logger.log('⚠️ 層6: シート書き戻しエラー（更新処理は継続）: ' + writeErr.toString());
  }
}
```

OR もし書き戻し実装が複雑すぎる場合は、ダイアログのテキストを「今回はeBay側の画像を使用する」に変更して期待値を合わせる。

### 問題2（中程度）: `use_sheet` 選択後もシートのURLが更新されず popup 再発

**問題**: ユーザーが `use_sheet` を選んで成功した場合、層1で新しいEPS URLが生成されるが、シートの該当列に書き戻されない。次回更新でもシートは lh3 URLのまま → 差分検出 → popup 再発。

**修正案**: `resumeReviseAfterDivergence()` で `reviseFixedPriceItem()` の戻り値が `success: true` だった場合、`normalizeImagesForRevise()` が生成した最終URLをシートに書き戻す。

実装が複雑になる場合の代替案: `reviseFixedPriceItem()` の成功時に `writeEpsUrlsToSheet()` を呼び出す（Add時と同じ）。ただし更新時の意図しない書き戻しに注意。

---

### 問題3（軽微）: `writeEpsUrlsToSheet()` のインデックスずれ (既存バグ)

`uploadAllImagesToEPS()` は失敗した画像を `epsUrls` から除外する（スキップ）が、`writeEpsUrlsToSheet()` は `epsIndex` をスキップなく常にインクリメントする。

```
例: 画像1=失敗, 画像2=成功 の場合
  originalImages = ["lh3_1", "lh3_2"]
  epsUrls        = ["i.ebayimg.com/EPS_2"]   ← 画像1が失敗のためスキップ
  
  writeEpsUrlsToSheet の動作:
    画像1列: originalUrl あり → epsUrls[0] = "i.ebayimg.com/EPS_2" を書き込む ← 画像2のURLが誤って画像1列に！
    画像2列: originalUrl あり → epsUrls[1] = undefined → 書き戻しなし（空欄のまま）
```

これが 69行目問題の遠因（画像1列に誤ったURLが書き戻された or ストア画像列だけEPS URLが残った）の可能性がある。修正は本PRスコープ外だが、将来のIssueとして記録推奨。
