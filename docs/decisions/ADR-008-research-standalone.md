# ADR-008: リサーチシート GAS のスタンドアロンライブラリ化

**日付:** 2026-05-10  
**ステータス:** 承認済み  
**決定者:** Shingo

## 背景

新規クライアント向けのリサーチシートを、出品ツール（`gas/listing/standalone/`）と同様に
スタンドアロンライブラリとして提供する。既存クライアント（コンテナバインド）は影響なし。

**調査レポート:** `docs/investigations/research-standalone-feasibility-20260510.md`

### 課題

現在の `gas/research/container/` はコンテナバインド方式のため、クライアントごとに
スクリプトを個別デプロイする必要がある。ライブラリ化することで：

- スクリプト1本を全クライアントで共有
- バグ修正・機能追加が即座に全クライアントに反映
- 新規クライアントのセットアップ工数削減

## 決定事項

### 1. OAuthトークンの保存先

**決定:** バインドスクリプト側の `ScriptProperties` に保存する。

**理由:** ライブラリの `PropertiesService.getScriptProperties()` はライブラリのスクリプトIDに
紐づく共有ストレージを返す。ライブラリ側にトークンを保存すると全クライアントが同一キーを
読み書きするため、クライアント間でトークンが混在するセキュリティリスクがある。

**実装方針:** ライブラリ関数はトークン文字列を引数として受け取る。
バインドスクリプト側が自身の `ScriptProperties` からトークンを取得して渡す。

### 2. LockService 戦略

**決定:** `LockService.getScriptLock()` → `LockService.getUserLock()` に変更する。

**理由:** ライブラリの `getScriptLock()` はライブラリレベルのロックを取得するため、
クライアントAがロックを保持している間はクライアントBもブロックされる。
`getUserLock()` はユーザー単位のロックであり、クライアント間の影響はない。

### 3. 移行対象

**決定:** **新規クライアントのみ**。既存クライアントはコンテナバインドを維持する。

**理由:** 既存クライアントへの影響をゼロにする。スタンドアロン版の品質が安定した後、
段階的に既存クライアントの移行を検討する。

## ディレクトリ構成（実装後）

```
gas/research/
├── container/    — 既存コンテナバインド（変更なし・既存クライアント継続使用）
├── standalone/   — スタンドアロンライブラリ（新規・新規クライアント向け）
├── bind/         — バインドスクリプトテンプレート（新規・新規クライアントが導入）
└── lowest-price/ — 最安値リサーチ（既存・変更なし）
```

## 却下した代替案

| 案 | 却下理由 |
|---|---|
| 全クライアントを一括移行 | 既存クライアントへのリスクが高い |
| トークンをスプレッドシートのシートに保存 | セキュリティリスク（シート閲覧者に見える） |
| getScriptLock() のまま維持 | クライアント間のクロスブロッキングが発生 |
